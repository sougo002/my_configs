# 共有レビュープロセス

このドキュメントは `review-pr`、`self-review` スキルから参照される共通のレビューロジックを定義する。
単体ではスキルとして機能しない。

## 前提条件

呼び出し元スキルが以下を準備すること:

- **作業ディレクトリ**: worktreeまたはリポジトリのパス
- **ベースブランチ名**
- **変更ファイル一覧**: `git diff --name-status <base>...HEAD` の出力
- **diff**: `git diff <base>...HEAD` の出力

## 並列レビュー（サブエージェント）

以下の4つのサブエージェント（Task ツール, subagent_type=general-purpose, model=opus）を起動する。
**重要: 4つの Task ツール呼び出しを必ず1つのレスポンス内でまとめて行うこと。これにより並列実行される。**

各サブエージェントには変更ファイル一覧・diff・参照ガイドのパスを渡す。
サブエージェントの作業ディレクトリは呼び出し元が指定した作業ディレクトリとする。
サブエージェントは必要に応じて Read で完全なソースコードや周辺コードを確認する。

| サブエージェント | 観点 | 参照ガイド |
|-----------------|------|-----------|
| アーキテクチャ | 設計原則、依存関係、コードスメル | [ARCHITECTURE.md](ARCHITECTURE.md) |
| セキュリティ | 脆弱性、認証、入力検証 | [SECURITY.md](SECURITY.md) |
| コード品質 | 可読性、構造、エラーハンドリング | [CHECKLIST.md](CHECKLIST.md) |
| テスト | カバレッジ、テスト品質 | [CHECKLIST.md](CHECKLIST.md) |

### サブエージェント指示テンプレート

呼び出し元スキルは、ガイドファイルのパスをスキルファイルの位置から `../review/` として解決し、
サブエージェントに絶対パスで渡すこと。

```
以下の変更をレビューしてください。出力は日本語で行うこと。

## 観点
[アーキテクチャ / セキュリティ / コード品質 / テスト]

## 参照ガイド
[対応するガイドファイルの絶対パスを Read で読み込んで基準として使用すること]

## 変更ファイル一覧
[git diff --name-status の出力]

## diff
[git diff <base>...HEAD の出力]

## レビュー範囲
**レビュー対象は diff に含まれる変更行のみとする。**
- 変更されていない既存コード（他の関数、テストケース、ファイル等）に対する指摘は行わないこと
- ただし、今回の変更によって新たに生じた問題（変更行が参照する値の不整合、変更が原因で発生するバグ等）は指摘すること
- 周辺コードの Read は、変更行の正しさを判断するための文脈理解にのみ使用すること

## 出力形式
重大度（Critical/High/Medium/Low）と共に所見を箇条書きで報告すること。
良い実装があれば併せて報告すること。
```

## 妥当性検証

サブエージェントから返された各指摘について、実際のコードを確認し妥当性を検証する。

各指摘に対して以下を行う:

1. **該当箇所の確認**: 指摘が参照するファイル・行を作業ディレクトリ上で Read し、指摘内容が実際のコードと一致するか確認する
2. **文脈の確認**: 必要に応じて周辺コード・呼び出し元・型定義などを確認し、指摘が正しい前提に基づいているか検証する
3. **判定**: 各指摘を以下のいずれかに分類する
   - **採用**: 指摘が妥当。そのまま結果に含める
   - **修正**: 指摘の方向性は正しいが、詳細や重大度に誤りがある。修正して含める
   - **却下**: 誤検知、または変更範囲外の既存コードへの指摘。結果から除外する

### 却下基準

以下に該当する指摘は除外する:

- 実際のコードと異なる前提に基づいている（存在しない変数・関数への言及、型の誤認など）
- diff に含まれない既存コードへの指摘（レビュー範囲の指示に反するもの）
- フレームワークやライブラリが既に保証している事項への冗長な指摘
- 同一の問題に対する重複した指摘（最も適切な1つに集約する）

## レビュー優先度

| 優先度 | 基準 | 例 |
|--------|------|-----|
| Critical | マージ前に必ず修正 | セキュリティ脆弱性、データ消失リスク、破壊的変更 |
| High | 修正すべき | 設計原則違反、エラーハンドリング欠如、パフォーマンスボトルネック |
| Medium | 修正を検討 | スタイル不統一、テスト不足、技術的負債 |
| Low | 提案 | 命名改善、ドキュメント不足、軽微なリファクタリング |
